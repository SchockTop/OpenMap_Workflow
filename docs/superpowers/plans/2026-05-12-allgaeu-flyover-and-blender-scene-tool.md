## Goal

1. **Deliverable A вҖ” a finished `.blend`** for the user's ~45 kmВІ AllgГӨu polygon (Forggensee/Schwangau/FГјssen): displaced DGM1 terrain, DOP40 ortho drape, LoD2 buildings textured from the ortho (good top-down look), trees that look good from any angle and concentrated on actual forest (not flat billboards, not scattered on lakes/town), volumetric clouds in the sky (fly-through-able, don't block the view), a cinematic fly-over camera, plus rendered hero stills. So `openmap_blender_tools` must have a clean **"Build cinematic scene from folder"** operator + N-panel UI: point it at a folder of downloaded tiles (or already-GDAL-processed data), pick a few options, get a finished cinematic scene. Add the **clouds** feature to that tool.

The GDAL preprocess (DGM mosaicвҶ’heightmap.tif, DOPвҶ’UDIM) happens *inside* `import_heightmap`/`import_ortho` (via `geo_import.py` shelling to vendored GDAL at `vendor/gdal-win64/bin/`).

- `features/trees.py` вҖ” appends real (decimated) 3D trees from `assets/trees.blend` (Oak/Beech/Spruce/Birch), scatters via Geometry Nodes `InstanceOnPoints` with a `density_mask` named attribute that currently defaults to 1.0 everywhere (no forest masking). - `world_setup.py` вҖ” `setup_multiple_scattering_sky` (Blender 5.1 Sky Texture), `add_domain_cube_volume` (aerial haze), `load_vdb_cloud` (needs an external `.vdb`). **No procedural clouds. No cloud assets.** вҶ’ GAP, we add it.
- `sky_presets.py`, `camera_presets.py`, `quality_presets.py`, `waypoint_generators.py`, `waypoints_to_camera.py`, `cli.py` (headless CLI, no full-pipeline command), `build_extension.py` (вҶ’ `dist/blender_tools-0.1.0.zip`, install via `blender --command extension install-file ... --repo user_default --enable`).

- Extension import path in Blender: `bl_ext.user_default.blender_tools` (extension must be installed). **Check whether `_blender_assemble_full.py` imports the installed extension or adds the repo source to `sys.path`** вҖ” determines whether code changes need a rebuild+reinstall before assembly.

## Decisions

- **Scope:** full ~45 kmВІ polygon, **DGM1** terrain + **DOP40** (40 cm) ortho + **LoD2** buildings. (40 cm is plenty for a fly-over; ~3вҖ“4 GB DOP40 vs ~15+ GB for DOP20. Pipeline's `muc-sued-10x4` preset is already 40 kmВІ, so 45 kmВІ is within range.) Subdiv 11 on the terrain plane (вүҲ4.4 m/quad over 9 km вҖ” fine from the air).
- **Clouds:** new `features/clouds.py` вҖ” procedural **volumetric** cumulus deck (a big thin box with a Principled-Volume material driven by layered noise: a "coverage" control, soft bottoms, slight anvil flattening on top), placed above the terrain (AllgГӨu terrain ~750вҖ“1800 m, peaks ~2000 m вҶ’ cloud base ~2200вҖ“2600 m absolute, in scene-local Z that's ~base minus anchor). Broken coverage so it doesn't block the view; tuned step size / not-too-dense for efficiency. Maybe an optional thin high cirrus layer. Operator `BLENDERTOOLS_OT_add_clouds` with props (coverage 0вҖ“1, base altitude, thickness, density, +cirrus toggle). Wired into the scene-builder + `_blender_assemble_full.py` `--enable clouds`.
- **Trees вҶ’ forest mask:** rasterize a forest mask for the AOI and feed it into `trees.py`'s `density_mask`. Best source = OSM land-use forest polygons (`OpenMap_Unifier/backend/osm_downloader.py` already produces `downloads_osm/land_use.geojson`) вҶ’ `gdal_rasterize` to a 0/1 GeoTIFF at the terrain's UV resolution вҶ’ sample as `density_mask` in the GN tree setup. Fallback if no OSM layer: an RGB "greenness" mask from the DOP (`ExG = 2GвҲ’RвҲ’B`, thresholded) via `gdal_calc`. Bump on-forest density, near-zero elsewhere. Also: nudge tree leaf shading toward "not obviously a card" вҖ” a touch of translucency/back-light on the leaf material (Translucent BSDF mix or Principled `Transmission`/`Subsurface` low) so backlit forest reads as a forest mass, not flat sprites.
- **Building look from above:** `--enable buildings-textured` (DOP-on-roofs) вҖ” already does what the user asked ("satellite mapped onto the buildings").
- **Workflow tool:** improve `BLENDERTOOLS_OT_quick_scene_from_folder` (or add `BLENDERTOOLS_OT_build_cinematic_scene` if cleaner) to be the one-stop folderвҶ’scene op: terrain + ortho + buildings + **building textures** + **forest-masked trees** + **clouds** + sky preset + camera preset + quality preset + (optional) camera-rig-from-waypoints. Accept either raw tiles *or* an already-GDAL-processed folder (`heightmap.tif` + `ortho_udim/` + `buildings.gml`/`.cityjson`) so the GDAL step can be done on machine A. Expose the key choices as operator properties (so the redo panel / N-panel works). Make sure the **N-panel "OpenMap" tab** lists this big button + the individual steps cleanly.

## Risks / fallbacks

- **45 kmВІ too heavy for assembly/render** вҶ’ reduce terrain subdiv, or split into a 3Г—3 tile grid of sub-scenes, or deliver a smaller hero area (Neuschwanstein/Hohenschwangau/Forggensee shore) at full quality + the wide area at lower detail.

- **Extension not installed / source-import path** вҶ’ `_blender_assemble_full.py` may import the installed extension; if so, rebuild+reinstall after the code changes (step 3) before assembly (step 4).
- **No NIR band for true NDVI** вҶ’ use OSM forest polygons (preferred) or RGB greenness (ExG) as the mask; documented.
- **Cycles + volumetric clouds slow at this scale** вҶ’ Eevee Next for renders; keep cloud density modest and step size tuned.
- **Context budget (main agent)** вҶ’ delegate the chunky work (downloads, GDAL, Blender assembly, rendering, review) to subagents that report summaries; keep this file as the source of truth across compactions.
