# Allgäu fly-over scene + Blender "build cinematic scene from folder" tool — plan

**Date:** 2026-05-12. **Owner:** main agent (subagent-driven execution).

## Goal

1. **Deliverable A — a finished `.blend`** for the user's ~45 km² Allgäu polygon (Forggensee/Schwangau/Füssen): displaced DGM1 terrain, DOP40 ortho drape, LoD2 buildings textured from the ortho (good top-down look), trees that look good from any angle and concentrated on actual forest (not flat billboards, not scattered on lakes/town), volumetric clouds in the sky (fly-through-able, don't block the view), a cinematic fly-over camera, plus rendered hero stills. Self-contained: data packed into the `.blend`. Lives in `workflows/scenes/allgaeu-flyover/` so it can be lifted out (with a manifest of which code commits belong to it).
2. **Deliverable B — the Blender tool fits the user's real workflow.** They're behind a proxy: maps are downloaded on machine A (OpenMap_Unifier GUI), dragged to machine B, scene built in Blender there. So `openmap_blender_tools` must have a clean **"Build cinematic scene from folder"** operator + N-panel UI: point it at a folder of downloaded tiles (or already-GDAL-processed data), pick a few options, get a finished cinematic scene. Add the **clouds** feature to that tool.

## What already exists (from the pipeline-map exploration — see that report)

- `workflows/full_pipeline.py` — download → GDAL → CityJSON → waypoints → Blender assemble (`_blender_assemble_full.py`, spawns Blender 5.1). **Only accepts named region presets, not KML.** → we add the polygon as a preset (done: `allgaeu-forggensee` in `region_presets.py`).
- `openmap_blender_tools/operators.py` — `quick_scene_from_folder` already chains import_heightmap → import_ortho → import_buildings → apply_sky → apply_quality → apply_ground_shader → scatter_trees → apply_camera_preset. **Missing from it:** building textures (`buildings_textured.apply`), clouds, camera-rig from waypoints. The GDAL preprocess (DGM mosaic→heightmap.tif, DOP→UDIM) happens *inside* `import_heightmap`/`import_ortho` (via `geo_import.py` shelling to vendored GDAL at `vendor/gdal-win64/bin/`).
- `features/buildings_textured.py` — DOP ortho box-projected onto roofs via a `DOPProjector` empty + procedural brick walls. Works; just not auto-enabled in `quick_scene_from_folder`.
- `features/trees.py` — appends real (decimated) 3D trees from `assets/trees.blend` (Oak/Beech/Spruce/Birch), scatters via Geometry Nodes `InstanceOnPoints` with a `density_mask` named attribute that currently defaults to 1.0 everywhere (no forest masking). Normal/roughness maps stripped → flat-ish leaf shading at close range (fine for aerial).
- `ndvi_scatter.py` — has `compute_ndvi` (gdal_calc) + `ndvi_to_density_config`, **not wired into `trees.py`**.
- `world_setup.py` — `setup_multiple_scattering_sky` (Blender 5.1 Sky Texture), `add_domain_cube_volume` (aerial haze), `load_vdb_cloud` (needs an external `.vdb`). **No procedural clouds. No cloud assets.** → GAP, we add it.
- `sky_presets.py`, `camera_presets.py`, `quality_presets.py`, `waypoint_generators.py`, `waypoints_to_camera.py`, `cli.py` (headless CLI, no full-pipeline command), `build_extension.py` (→ `dist/blender_tools-0.1.0.zip`, install via `blender --command extension install-file ... --repo user_default --enable`).
- `OpenMap_Unifier/backend/geometry.py::PolygonExtractor.extract_from_kml` — KML → WGS84 EWKT (used by the web app, not the pipeline).
- Extension import path in Blender: `bl_ext.user_default.blender_tools` (extension must be installed). **Check whether `_blender_assemble_full.py` imports the installed extension or adds the repo source to `sys.path`** — determines whether code changes need a rebuild+reinstall before assembly.

## Decisions

- **Scope:** full ~45 km² polygon, **DGM1** terrain + **DOP40** (40 cm) ortho + **LoD2** buildings. (40 cm is plenty for a fly-over; ~3–4 GB DOP40 vs ~15+ GB for DOP20. Pipeline's `muc-sued-10x4` preset is already 40 km², so 45 km² is within range.) Subdiv 11 on the terrain plane (≈4.4 m/quad over 9 km — fine from the air).
- **Clouds:** new `features/clouds.py` — procedural **volumetric** cumulus deck (a big thin box with a Principled-Volume material driven by layered noise: a "coverage" control, soft bottoms, slight anvil flattening on top), placed above the terrain (Allgäu terrain ~750–1800 m, peaks ~2000 m → cloud base ~2200–2600 m absolute, in scene-local Z that's ~base minus anchor). Broken coverage so it doesn't block the view; tuned step size / not-too-dense for efficiency. Maybe an optional thin high cirrus layer. Operator `BLENDERTOOLS_OT_add_clouds` with props (coverage 0–1, base altitude, thickness, density, +cirrus toggle). Wired into the scene-builder + `_blender_assemble_full.py` `--enable clouds`.
- **Trees → forest mask:** rasterize a forest mask for the AOI and feed it into `trees.py`'s `density_mask`. Best source = OSM land-use forest polygons (`OpenMap_Unifier/backend/osm_downloader.py` already produces `downloads_osm/land_use.geojson`) → `gdal_rasterize` to a 0/1 GeoTIFF at the terrain's UV resolution → sample as `density_mask` in the GN tree setup. Fallback if no OSM layer: an RGB "greenness" mask from the DOP (`ExG = 2G−R−B`, thresholded) via `gdal_calc`. Bump on-forest density, near-zero elsewhere. Also: nudge tree leaf shading toward "not obviously a card" — a touch of translucency/back-light on the leaf material (Translucent BSDF mix or Principled `Transmission`/`Subsurface` low) so backlit forest reads as a forest mass, not flat sprites.
- **Building look from above:** `--enable buildings-textured` (DOP-on-roofs) — already does what the user asked ("satellite mapped onto the buildings").
- **Workflow tool:** improve `BLENDERTOOLS_OT_quick_scene_from_folder` (or add `BLENDERTOOLS_OT_build_cinematic_scene` if cleaner) to be the one-stop folder→scene op: terrain + ortho + buildings + **building textures** + **forest-masked trees** + **clouds** + sky preset + camera preset + quality preset + (optional) camera-rig-from-waypoints. Accept either raw tiles *or* an already-GDAL-processed folder (`heightmap.tif` + `ortho_udim/` + `buildings.gml`/`.cityjson`) so the GDAL step can be done on machine A. Expose the key choices as operator properties (so the redo panel / N-panel works). Make sure the **N-panel "OpenMap" tab** lists this big button + the individual steps cleanly.

## Sequence (subagent-driven; main agent coordinates)

1. **[done] Add `allgaeu-forggensee` region preset** (`workflows/region_presets.py`).
2. **[long-running, kick off ASAP, background] Download + GDAL preprocess** the AOI: `python workflows/full_pipeline.py --region allgaeu-forggensee --datasets dgm1 dop40 lod2 --quality draft --camera-preset cinematic-establishing`. This downloads ~50 DGM1 + ~50 DOP40 + ~12 LoD2 tiles (~4–5 GB), GDAL-mosaics → `data/processed/heightmap.tif` + `data/processed/ortho_udim/` + `data/processed/buildings.cityjson`, generates a flight path, and (best-effort) draft-assembles a baseline `.blend`. If phase 5 (Blender) chokes at 45 km² or the extension isn't installed, phases 1–3 still leave the processed data on disk → proceed from there. Also download OSM land-use for the AOI for the forest mask (`OpenMap_Unifier/backend/osm_downloader.py` for the same polygon → `downloads_osm/land_use.geojson`).
3. **[parallel with #2] `openmap_blender_tools` improvements** (one subagent, cohesive — touches `operators.py` once): (a) new `features/clouds.py` + `BLENDERTOOLS_OT_add_clouds` + register + N-panel button + a `smoke_clouds.py`; (b) wire `buildings_textured` + clouds into `quick_scene_from_folder` (or new `build_cinematic_scene`) and expose options as props; accept pre-processed folders; (c) forest-mask hook in `trees.py` (`density_mask` from a mask GeoTIFF arg) + a `geo_import`/`ndvi_scatter` helper to rasterize OSM land-use forest → mask GeoTIFF; (d) leaf-shading tweak for the trees; (e) tests/smoke for the new bits; (f) `--enable clouds` in `_blender_assemble_full.py`. Then `python openmap_blender_tools/build_extension.py` → new `dist/blender_tools-0.1.0.zip`; reinstall in Blender 5.1 (`& "C:\Program Files\Blender Foundation\Blender 5.1\blender.exe" --command extension install-file dist/blender_tools-0.1.0.zip --repo user_default --enable`). Run `smoke_dommesh.py`-style smokes + `smoke_full_scene.py` to confirm nothing broke.
4. **Assemble the Allgäu scene** with the new features: `python workflows/full_pipeline.py --region allgaeu-forggensee --skip-download --local-dgm data/raw/... --local-dop ... --local-lod2 ... --enable buildings-textured trees clouds --camera-preset cinematic-establishing --quality preview --render-preview` (or drive the new operator headless). Produces `data/scene_allgaeu-forggensee.blend` + a preview render.
5. **Render hero stills** — 3–4 frames along the flight path (and maybe a couple from `aircraft-approach` / `mid-drone`), `preview` quality first, then `final` (1920×1080) for the keepers. Eevee Next if Cycles + volumetric clouds is too slow at 45 km².
6. **Independent review** — `/review-renders` skill on the PNGs + a fresh agent verifying: terrain reads as the real Allgäu relief, ortho looks right (no seams), buildings have ortho roofs, trees concentrated on forest and not obviously flat, clouds visible but not blocking, camera framing good. List issues.
7. **Iterate** on whatever the review flags (cloud density/coverage, tree mask threshold, terrain subdiv, camera altitude/path, building texture blend, sky preset/time-of-day).
8. **Final render + pack + deliver:** `bpy.ops.file.pack_all()` → save to `workflows/scenes/allgaeu-flyover/scene.blend`; copy the hero PNGs + a `README.md` (how it was made, how to re-render, which datasets, the polygon) + `MANIFEST.md` (every commit/file added for this) into that folder. Commit. Push (parent + submodule bumps, per the repo's submodule workflow).
9. **Hand-off note** — tell the user: the updated `openmap_blender_tools` extension zip is at `dist/blender_tools-0.1.0.zip`; on the Blender machine, install it; the new "Build cinematic scene from folder" button in the OpenMap N-panel takes a folder of downloaded tiles (or pre-processed data) → finished scene with clouds. Also note the GDAL dependency for the raw→processed step (run on the machine that has the repo's `vendor/gdal-win64/`, or have GDAL/QGIS on PATH on the Blender machine).

## Risks / fallbacks

- **45 km² too heavy for assembly/render** → reduce terrain subdiv, or split into a 3×3 tile grid of sub-scenes, or deliver a smaller hero area (Neuschwanstein/Hohenschwangau/Forggensee shore) at full quality + the wide area at lower detail.
- **Downloads fail on the proxy network** → patch `full_pipeline.py` phase 1 (or the `MapDownloader` call) to use `OpenMap_Unifier.backend.proxy_manager.get_proxy_manager().get_session()` like the dommesh fix did.
- **Extension not installed / source-import path** → `_blender_assemble_full.py` may import the installed extension; if so, rebuild+reinstall after the code changes (step 3) before assembly (step 4).
- **No NIR band for true NDVI** → use OSM forest polygons (preferred) or RGB greenness (ExG) as the mask; documented.
- **Cycles + volumetric clouds slow at this scale** → Eevee Next for renders; keep cloud density modest and step size tuned.
- **Context budget (main agent)** → delegate the chunky work (downloads, GDAL, Blender assembly, rendering, review) to subagents that report summaries; keep this file as the source of truth across compactions.
